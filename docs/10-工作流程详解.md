# 工作流程详解：用 OCP Catalog 包阿里妈妈

> **这份文档只做一件事**：把整个系统讲清楚，让你看完能在白板上重画出来。
> 不讲怎么开发、不讲怎么部署、不讲怎么汇报——**只讲它是怎么运行的**。
> 配合 [09-执行手册.md](09-执行手册.md) 阅读最佳：那份让你跑起来，这份让你懂。

---

## 一句话本质

**写一个翻译服务，把"AI Agent 说 OCP 协议"和"淘宝联盟说 Alimama API"互译。**

```text
   AI Agent                  alimama-provider                Alimama API
   ━━━━━━━━━                ━━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━
   说 OCP 协议    ⇄ 翻译 ⇄    说两种语言        ⇄ 翻译 ⇄   说阿里 API

   不知道阿里                同时懂两边                     不知道 OCP
```

中间那个 `alimama-provider`，就是这次 Day 1-7 我们写的核心服务。

---

## 三方角色清单

| 系统 | 它在干嘛 | 它**不**干嘛 | 关键文件 |
| --- | --- | --- | --- |
| **AI Agent** | 用 OCP 协议查商品、给用户推荐 | 不直接调阿里 API、不懂 PID | （未来接 deeplumen 自家 Agent） |
| **OCP Catalog** | 协议门面 + 检索引擎；接收 sync 入库、接 query 检索、resolve 时回调 Provider | 不知道阿里、不存 PID | `OCP-Catalog/apps/examples/commerce-catalog-api/` |
| **alimama-provider** ★ | 翻译适配：拉阿里物料、转 OCP 格式、回调时调阿里转链 | 不存商品索引、不做检索 | `OCP-Catalog/apps/examples/alimama-provider-api/` |
| **阿里妈妈** | 商品库 + 链接工厂 + 佣金账本 | 不懂 OCP、不知道 Agent 存在 | （外部 HTTP API） |

> **技术上 100% 等同于"写好那个 alimama-provider"。** OCP-Catalog 我们只改了 90 行（resolve 回调机制），阿里妈妈我们完全不动。

## 系统启动时发生什么

按 [09-执行手册.md](09-执行手册.md) 跑完前 3 步后，**三个进程同时运行**：

```text
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  Postgres :55432           commerce-catalog-api :4000              │
│  ━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━                  │
│  ocp-catalog-postgres-wsl  连 :55432, 提供 /ocp/* 接口              │
│  (Docker 容器,内含数据)    (我们改了 commerce-scenario 90 行)       │
│                                                                     │
│                                                                     │
│  alimama-provider-api :4300                                         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━                                           │
│  连 :4000 (catalog)、连 gw.api.taobao.com (alimama)                │
│  暴露 /admin/* (运营接口) 和 /provider/resolve_hook (给 catalog 回调)│
│  可选: 启动两个 cron worker(MATERIAL_POLL + ORDER_POLL)             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

启动顺序固定：**Postgres → catalog → provider**。
理由：catalog 启动要读 DB；provider 启动会注册到 catalog，要 catalog 先就位。

## 四个核心工作流

整个系统就 4 个流。每个都一遍想清楚后，整套系统就懂了。

### 工作流 1 — 商品同步（让淘宝商品进 OCP）

**触发方式**：手动 `POST /admin/sync` 或 cron 自动定时。

**流程**：

```text
[运营/cron]
  │ POST /admin/sync {q: "耳机", pageSize: 6}
  ▼
[alimama-provider :4300]
  │
  │ 1) 调阿里
  │    POST gw.api.taobao.com/router/rest
  │    method=taobao.tbk.dg.material.optional
  │    q=耳机, adzone_id=116271650208, page_size=6
  ▼
[阿里妈妈]
  │ 返回 6 个商品(JSON, snake_case, 中文 title, 价格字符串等)
  ▼
[alimama-provider 的 mapper]
  │ 把每个 AlimamaMaterialItem 翻译成 OCP CommercialObject:
  │   - num_iid       → object_id
  │   - title         → product.core.v1.title
  │   - pict_url      → 加 https: 前缀 → image_urls[0]
  │   - small_images  → image_urls[1..]
  │   - zk_final_price → 字符串 parseFloat → price.v1.amount
  │   - user_type=1   → attributes.platform = "tmall"
  │   - commission_rate → attributes.commission_rate_bp
  │   - coupon_*      → attributes.coupon = {...}
  │   - 加 attributes.requires_affiliate_resolution = true ★
  │   - 加 attributes.provider_resolve_hook_url = "http://localhost:4300/..." ★
  ▼
[alimama-provider 把翻译好的 CommercialObject 批量推给 catalog]
  │ POST /ocp/objects/sync
  │ batch: 6 objects(自动按 100 拆批,这次没到上限)
  ▼
[commerce-catalog-api :4000]
  │ 1) 校验每个 CommercialObject 满足 ObjectContract
  │ 2) 跑 buildSearchProjection(object) 把字段平铺成索引
  │    ⚠️ 关键: 我们 Day 4 改了这个函数,让 attributes 也进 projection
  │ 3) 写 3 张表:
  │    - commercial_objects(原始 OCP 对象)
  │    - descriptor_instances(每个 descriptor pack 一行)
  │    - catalog_entries(索引投影,供搜索用)
  ▼
[Postgres :55432]  数据落地
```

**返回**：

```json
{
  "total": 6, "batches": 1,
  "results": [{
    "status": "accepted",
    "accepted_count": 6, "rejected_count": 0,
    "items": [{ "object_id": "...", "catalog_entry_id": "centry_..." }, ...]
  }]
}
```

**关键设计点**：
- **mapper 处理鲁棒性**：图片 URL 缺 scheme、price 是字符串、字段可能缺失——`material-to-object.ts` 都兜底
- **`attributes.provider_resolve_hook_url`**：在物料同步阶段就埋进商品的 attributes，等 resolve 时 catalog 知道回调谁

### 工作流 2 — Agent 查询商品（OCP 的发现层）

**触发方式**：Agent 调 `POST /ocp/query`。这是**纯 catalog 内部**的流程，**provider 完全不参与**。

```text
[AI Agent]
  │ POST :4000/ocp/query
  │ {
  │   query_pack: "ocp.query.keyword.v1",
  │   query: "耳机",
  │   filters: { provider_id: "alimama_real" },
  │   limit: 10
  │ }
  ▼
[commerce-catalog-api]
  │ 1) 验证 query_pack 是支持的
  │ 2) 按 filter 过滤 + keyword 做 pg_trgm 模糊匹配(title/text 字段)
  │ 3) 排序 + 截取 limit
  ▼
[Postgres]
  │ SELECT * FROM catalog_entries
  │ WHERE provider_id='alimama_real'
  │   AND (title ILIKE '%耳机%' OR text ILIKE '%耳机%')
  │ ORDER BY ...
  │ LIMIT 10
  ▼
[catalog 返回 Agent]
  │ {
  │   result_count: 6,
  │   items: [{
  │     entry_id: "centry_...",
  │     title: "...",
  │     score: ...,
  │     summary: "..."
  │   }, ...]
  │ }
```

**关键设计点**：
- **alimama-provider 完全不出现在这条链路上** —— 商品已经入库了，查询用本地 catalog 就够
- **catalog 返回的字段是经过 `buildSearchProjection` 处理的子集**（不会暴露 PID、rfq_url 这些敏感 attributes，得 resolve 时才出）

---

### 工作流 3 — Resolve 转链（⭐ 整个系统的高潮）

**触发方式**：Agent 选定一个商品，调 `POST /ocp/resolve`。这是**催生 affiliate URL 的关键时刻**——Day 4 的 90 行 patch 就是为这一步存在的。

```text
[AI Agent]
  │ POST :4000/ocp/resolve
  │ {
  │   entry_id: "centry_...",
  │   agent: { agent_id: "agt_demo_001", intent: "shopping_purchase" }  ★ 协议扩展
  │ }
  ▼
[commerce-catalog-api]
  │
  │ 1) 查 catalog_entries 表拿 projection(含 attributes)
  │
  │ 2) 跑 commerce-scenario.buildResolveActions(projection, ctx):
  │    A. 先加静态 view_product action(用原始 item_url,无 PID)
  │    B. 看 projection.attributes.requires_affiliate_resolution
  │       如果为 true,且 attributes.provider_resolve_hook_url 存在:
  │
  │       ┌─────────────────────────────────────────────────┐
  │       │ POST attributes.provider_resolve_hook_url      │
  │       │ {                                                │
  │       │   entry_id, object_id,                           │
  │       │   agent_id ← ctx.request.agent.agent_id          │
  │       │ }                                                │
  │       │ 超时 2 秒,失败就降级到只返 view_product           │
  │       └─────────────┬───────────────────────────────────┘
  │                     │
  │                     ▼
  │           [alimama-provider :4300]
  │             │ /provider/resolve_hook
  │             │
  │             │ 1) 查 adzone_id(PoC 阶段固定一个,
  │             │    生产阶段按 agent_id 路由不同 adzone)
  │             │
  │             │ 2) 调阿里转链:
  │             │    POST gw.api.taobao.com/router/rest
  │             │    method=taobao.tbk.privilege.get
  │             │    item_id=<object_id>
  │             │    adzone_id=116271650208
  │             │    external_id=<entry_id>   ★ 透传到订单回执
  │             │
  │             ▼
  │           [阿里妈妈]
  │             │ 返回 {
  │             │   coupon_click_url: "https://s.click.taobao.com/...",
  │             │   item_url: "https://s.click.taobao.com/...",
  │             │   mm_coupon_click_url: "...",      // 天猫专属券
  │             │   coupon_info: "满 99 元减 10 元"
  │             │ }
  │             │
  │             ▼
  │           [alimama-provider 的 privilege-to-action mapper]
  │             │ 翻译成 OCP ActionBinding[]:
  │             │   - buy_with_coupon  → coupon_click_url
  │             │   - buy_with_tmall_coupon → mm_coupon_click_url(if exists)
  │             │
  │             │ 返回 { action_bindings: [...] }
  │             ▼
  │       ◀────┘
  │
  │ 3) 合并 view_product + provider 返回的 bindings,组装 ResolvableReference
  ▼
[catalog 返回 Agent]
  │ {
  │   title: "...",
  │   action_bindings: [
  │     { action_id: "view_product", url: "https://detail.tmall.com/..." },
  │     { action_id: "buy_with_coupon", url: "https://s.click.taobao.com/..." }, ★
  │     { action_id: "buy_with_tmall_coupon", url: "https://s.click.taobao.com/..." }
  │   ],
  │   visible_attributes: { coupon: {...}, commission_rate_bp: 1550, ... },
  │   freshness: { resolved_at: "..." },
  │   expires_at: "..."     // 15 分钟 TTL
  │ }
```

**关键设计点**：
- **协议层扩展 1**：`ResolveRequest` 加 `agent` 字段（Day 4 Patch 001）。允许 Agent 携带身份。
- **协议层扩展 2**：`Scenario.buildResolveActions` 改 async + 加 `ResolveContext`（Day 4 Patch 002+003）。允许 catalog 回调外部 HTTP。
- **协议层扩展 3**：`buildSearchProjection` 透传 `attributes`（Day 4 Patch 005）。否则 hook URL 到不了。
- **降级路径**：provider 挂了 / 阿里慢了，catalog 仍然返回 view_product fallback，Agent 拿到的体验是"链接还能用、只是没有 PID 归因"
- **`external_id` 透传**：把 OCP entry_id 塞进 alimama 转链请求，订单回执里会带回来——这是日后做 entry-level attribution 的钩子

### 工作流 4 — 用户成交 + 佣金回流（异步，跨天）

这是**资金落地**的链路，不在 Agent 同步链路上。

**第 1 阶段：用户实际购买**（OCP 系统完全不参与）

```text
[用户] 点击 ActionBinding 里那个 s.click.taobao.com/xxx 短链
  │
  ▼
[阿里 s.click 服务]
  │ 解析 PID,在用户浏览器/手淘 App 写入 cookie:
  │   pid=mm_10244555183_3415750080_116271650208
  │   external_id=centry_...
  │
  │ 302 跳转到真实商品页 detail.tmall.com/item.htm?id=xxx
  ▼
[淘宝商品页 / 手淘 App]
  │ 用户加购、付款
  ▼
[淘宝订单系统]
  │ 订单成立,把 PID 关联起来
  │ tk_status = 12(已付款,佣金预估中)
  ▼
[等用户确认收货,3-15 天]
  │ tk_status = 13(已结算,佣金到账)
  │ 阿里联盟把佣金扣 10% 技术服务费后,记到淘客账户余额
```

**第 2 阶段：alimama-provider 的 order-poller 拉回执**（cron 触发）

```text
[cron 每 20 秒]
  │
  ▼
[alimama-provider 的 order-poller]
  │ 调阿里
  │   POST gw.api.taobao.com/router/rest
  │   method=taobao.tbk.order.get
  │   start_time=<24 小时前>
  │   end_time=<now>
  │   query_type=pay_time
  ▼
[阿里妈妈]
  │ 返 [{ trade_id, item_id, tk_status, alipay_total_price,
  │       pub_share_pre_fee, adzone_id, ... }, ...]
  ▼
[order-to-ledger mapper]
  │ tk_status 12 → "paid"
  │ tk_status 13 → "settled"
  │ tk_status 14 → "invalid"
  │ tk_status 15 → "under_dispute"
  │ alipay_total_price 字符串 → 整数(分)
  │ adzone_id 116271650208 → agent_id  (PoC 阶段还没做反向 lookup,生产要做)
  ▼
[CommissionLedger(内存 Map,upsert 幂等)]
  │ key=trade_id, 同一订单状态变化时刷新而不重复入库
```

**第 3 阶段：随时查统计**

```text
[运营/老板]
  │ GET /admin/stats
  ▼
[alimama-provider]
  │ ledger.stats() 聚合:
  │   - 总订单数 / 按状态分布
  │   - 总成交金额 / 总预估佣金 / 总实际佣金
  │   - 按 adzone(可映射回 agent_id)分布
  ▼
[返回 JSON]
  {
    "total_orders": 4,
    "by_status": { "paid": 1, "settled": 1, ... },
    "total_pay_amount_fen": 53280,        // 532.80 元
    "total_estimated_commission_fen": 8921, // 89.21 元
    "total_real_commission_fen": 2777,    // 27.77 元(只 settled 的计入)
    ...
  }
```

**关键设计点**：
- **状态机一定要支持回退**：tk_status 13→15（已结算后维权）会扣回佣金，ledger 设计成 upsert 而非 append-only
- **内存版 ledger 是 PoC 妥协**：进程重启数据丢失。生产要换成 Postgres + Drizzle
- **`agent_id` 反向 lookup 还没实现**：当前 ledger 只记 adzone_id，要做"per agent commission stats"还要加 `agent_adzone_mapping` 表

## 几个"为什么"

### 为什么 catalog 要在 resolve 时回调 provider，而不是 sync 时就把 affiliate URL 入库？

**因为 PID 是动态的**。

- 静态思路：sync 时一次性算好 affiliate URL，存进 catalog。简单。
- 但**所有 Agent 共用一个 PID 会被阿里风控**（"异常重复使用"红线之一）
- 所以未来要做"一 Agent 一 PID"，必须**等 resolve 时知道 agent_id 才能选合适的 PID 生成链接**
- 这就是为什么 Day 4 改了 90 行让 catalog 支持 resolve-time 回调

PoC 阶段我们其实只有一个 adzone，但架构已经为 per-agent 准备好了。

### 为什么 ledger 用内存 Map 不用 Postgres？

**因为 PoC 不想引入 DB 迁移负担**。

ledger 表的访问模式是"按 trade_id upsert + 按时间窗口聚合"，接口设计成跟未来 Drizzle 版本一致（`upsert()` / `stats()` / `list()`）。等真上生产，换 4 个文件就完事。

进程重启数据丢失这个缺点对 PoC 演示没影响——演示前手动跑一次 `/admin/sync-orders` 就能重灌。

### 为什么 alimama-provider 不直接对 Agent 暴露接口？

**因为协议要稳定，Agent 不应该感知后端是阿里还是京东**。

未来加京东联盟 / 拼多多多多客：

```text
[Agent]
   │ OCP 协议 (始终不变)
   ▼
[OCP Catalog]
   │ resolve_hook 根据 attributes.affiliate_provider 路由
   ├──▶ alimama-provider   (taobao.tbk.*)
   ├──▶ jd-union-provider  (jd.union.*)
   └──▶ pdd-ddk-provider   (pinduoduo.ddk.*)
```

Agent 端零修改。这是 OCP 协议价值的核心：**它是 Agent 和具体联盟之间的稳定隔离层**。

### 为什么 image_urls 要绝对化、price 要 parseFloat？

**因为 OCP 协议比阿里 API 严格**。

阿里返：
- `pict_url: "//gw.alicdn.com/imgextra/..."` (无 scheme)
- `zk_final_price: "199.00"` (字符串)
- `commission_rate: 1550` (基点)

OCP 要求：
- `image_urls: [URL string]` (必须 https://)
- `price.amount: number` (必须 number)
- 不允许多余字段（descriptor schema 是 `.strict()`）

mapper 必须做这些转换。这就是 Day 1 的核心工作。

## 文档之间的关系

```text
00-overview           ← 5 分钟懂这事是干嘛(对外汇报版)
01-research           ← 战略 / 合规 / 市场背景(深)
02-technical-design   ← 设计稿(为什么这么写)
03-workflow           ← 原计划草案(Day 0 到 Week 5+)
04-decisions          ← 12 项决策清单
05-implementation     ← 实现报告(三方角色 + 代码片段)
06-coding-flow        ← Day 1-7 编码圣经(精确到 Step)
07-接入申请清单       ← 给业务方:如何申请阿里账号
08-完成版汇报         ← 完成后:5 分钟当面演示脚本 + 10 分钟自助
09-执行手册           ← 拿着这份你能复制粘贴跑通整套
10-工作流程详解       ← 你正在看 - 让你能在白板重画的那一份
```

读什么取决于你想干什么：

- 看一遍懂这事 → **00**
- 准备汇报 / 演示 → **08**
- 自己跑通 → **09**
- 在白板讲给别人 → **10（这份）**
- 接着写代码 → **06**
- 申请账号 → **07**
