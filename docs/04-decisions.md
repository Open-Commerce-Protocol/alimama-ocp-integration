### D2. "第三方 Agent" 的合规边界

**问题**：阿里妈妈红线"API 权限不得开放给第三方调用"，OCP 上的 Agent 算不算第三方？

**影响**：**这条直接决定 OCP 走开放协议还是封闭生态**。
- 如果 Agent 必须是同主体下的内部渠道 → OCP 实际只能服务自家 Agent
- 如果可以是签了协议的合作方 → OCP 可以做平台
- 如果可以是任意第三方 → OCP 可以做开放标准（最危险）

**可选**：
- A. **封闭**：只接 deeplumen 自家 Agent，零外部 Agent
- B. **白名单**：签协议的合作方 Agent 可接，需阿里妈妈书面认可
- C. **开放**：任意 Agent 可接，把 PID 暴露给协议层（违反红线，不推荐）

**推荐**：**A 起步、B 长期**，C 不可碰。

**拍板人**：法务 + 业务负责人

**状态**：🟡 **已转交业务**（见 [07-接入申请清单.md 决策 1](07-接入申请清单.md)）。法务/合规叙事在 07 文档 §法务 FAQ 已就绪，等业务签字。

**备注**：需要找阿里妈妈商务沟通 B 的可行性。

### D3. 是否接二房东 API 兜底

- 不接：阶段三只能跑到"展示 + 跳转"，不能归因佣金
- 二房东本身合规性也模糊（他们持有官方权限，把权限"租"给我们）

**可选**：

- B. **接维易**（功能多一些）
- C. **不接**，等官方权限

**推荐**：**D**。PoC + 演示阶段接订单侠（按调用量付费、月 ¥200-500 起），正式生产前切官方权限。

**拍板人**：业务 + 技术

**状态**：🟡 **已转交业务**（见 [07-接入申请清单.md 决策 2](07-接入申请清单.md)）。推荐姿态调整为：**Day 7 演示先用裸链不归因（决策 2.A）**，正式生产前再评估是否接二房东。

### D5. ResolveContext 把 fetch 注入是否安全

**问题**：scenario 拿到 `ctx.fetch` 后可以发任意外部请求。catalog 信任 scenario 的程度有多高？

**影响**：
- 注入：灵活，scenario 可以 mock；但需要约束 fetch 的目标 URL 来源
- 不注入：scenario 用全局 fetch，无法 mock

**可选**：
- A. 注入 fetch，且约束 `provider_resolve_hook_url` 必须 https + 域名白名单
- B. 不注入 fetch，由 scenario 自己创建
- C. 注入但加超时和重试默认值

**推荐**：**A + C**：注入有约束 fetch（超时 2s、固定 retry policy、目标域名白名单），白名单从 catalog manifest 读

**拍板人**：技术架构师

**状态**：✅ **已实现 A + 部分 C**（Day 4 Patch 002）。当前注入了 fetch + 2s `AbortSignal.timeout`,**未实现域名白名单**(Day 7+ TODO)。

## 阶段三决

### D6. 每 Agent 一 adzone vs 共用 adzone

**问题**：PoC 阶段所有 Agent 共用一个 adzone；什么时候切到 per-agent？

**影响**：
- 共用：实现简单；但归因粒度粗、PID 风控压力大
- per-agent：归因精细；但需要 adzone 池和分配逻辑

**可选**：
- A. 始终共用
- B. PoC 阶段共用，并发 ≥ 10 个 Agent 时切 per-agent
- C. 一上来就 per-agent

**推荐**：**B**。阶段一二三共用，阶段四观察风控信号决定，阶段五正式实现 per-agent。

**拍板人**：技术负责人

**状态**：🟡 **PoC 共用单 adzone**（env `ALIMAMA_ADZONE_ID=mock_adzone_001`)。per-agent 切换时机延后到 Day 7 真实流量观察后再定。
